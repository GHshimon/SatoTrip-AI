# SatoTrip 健康診断レポート

**診断日**: 2026-07-09
**対象コミット**: `e1ded1d`
**診断範囲**: フロントエンド（React + Vite）／バックエンド（FastAPI）／セキュリティ・商用化準備
**方式**: 読み取り専用のコード精査 + バックエンド実起動確認（`GEMINI_API_KEY` 等の実キーは未設定のため、AI生成・ホテル検索の実通信は未検証）

## 総評

アーキテクチャの土台（SQLAlchemy ORM によるSQLインジェクション耐性、bcryptパスワードハッシュ、所有者スコープ付きクエリ、Reactの自動エスケープ）は**概ね健全**です。一方で、**「商用リリースの即時ブロッカー」がフロント・バック・セキュリティの3領域すべてに存在**しており、現状は「デモは動くが、そのまま公開すると事故る」段階です。

完成度を上げるうえでの本質的な問題は3つに集約されます。

1. **秘密情報がGitにコミットされている**（JWT署名鍵・管理者パスワード） → 認証を完全にバイパスできる致命的欠陥
2. **開発中のデバッグコードが本番コードに残置**（`fetch('http://127.0.0.1:7243/...')` や `c:\projects\...\debug.log` への書き込みが**フロント19箇所・バック70箇所以上**） → 動作不良・情報漏洩・無駄な通信
3. **画面とAPIの「つなぎ込み」が未完成**（モックデータ参照の残存、ID体系の不一致、実装されていないボタン多数） → 「押しても何も起きない」ボタンが多発

以下、重要度順にすべて列挙します。件数は絞っていません。

| 重要度 | 件数 | 主な内容 |

## 🔴 致命的（公開前に必ず対応）

### C-1. 実運用のJWT署名鍵がGitにコミットされている
- **場所**: `backend/.env.production:2`（`JWT_SECRET_KEY=b4f0…be48`、64桁hexの実値）
- **症状/悪用**: この鍵を知る者は任意の `user_id`／`role` を含むJWTを自己署名でき、**認証を完全にバイパスして任意ユーザー・管理者になりすませる**。ルート直下に `.gitignore` が無いため追跡対象になっており、履歴にも残存。
- **原因**: `backend/.gitignore` は `.env` を無視するが、`.env.production` は別名のため無視から漏れた。
- **対応**: ①鍵をローテーション ②`.env.production` を追跡解除 ③`git filter-repo` 等で履歴から除去 ④シークレットは環境変数／シークレットマネージャ経由のみに。

### C-2. 管理者の平文パスワードがコミットされている
- **場所**: `login.json:1`（`{"username":"Admin_user001","password":"Teat…#001"}`）
- **症状/悪用**: `set_admin.py` が同ユーザーを管理者化するため実アカウントと一致。**誰でもこの資格情報で管理者ログイン可能**（ユーザー一覧・ロール変更・全設定・システムプロンプト改変・APIキー発行）。
- **対応**: ①当該管理者パスワードを即時変更 ②`login.json` を削除＋履歴除去 ③管理者初期化は対話的／環境変数で行う。

### 機能が壊れている / 危険（バックエンド）

**H-11. プラン更新エンドポイントが無検証入力で500になる**
- `app/api/plans.py:1051` — `PUT /api/plans/{id}` に `spots` が `id` を欠く辞書を含むと `KeyError:'id'` → 500。実際に `{"spots":[{"startTime":"10:00"}]}` で再現確認済み。
- **修正**: `.get("id")` を使い、欠落要素はスキップ or 422。

**H-12. プラン生成の使用量が「失敗しても」消費される**
- `app/api/plans.py:913-917` — `record_plan_generation()` を `create_plan()` の**前**に呼ぶため、生成が例外で失敗しても月間使用回数だけ加算。実測で失敗1回後に remaining 5→4。
- **修正**: 使用量記録を成功後に移動。

**H-13. folder_id / parent_id に所有権検証がなくIDOR**
- `app/services/folder_service.py`（update_folder）/ `plan_service.py:149` — 他ユーザーのフォルダIDを指定でき、実測で user2 が user1 のフォルダ配下に作成成功・user1 のプランを user2 のフォルダへ割当成功。
- **修正**: 指定時に `get_folder(db, id, current_user.id)` で所有権確認。

**H-14. Windowsデバッグパスへの大量ログ書き込み（バック70箇所以上）**
- `app/dependencies.py:74`、`app/api/spots.py:338`、各 service、`utils/subscription.py` 等8ファイル — `open(r'c:\projects\SatoTrip-AI\.cursor\debug.log','a')` 形式のコードが散在。Linuxでは `c:\projects\...` というゴミファイルがCWDに生成される（実際に確認・削除済み）。APIキー先頭・フォルダ名・ユーザーIDを書き出しており情報漏洩リスク。
- **修正**: 全「#region agent log」ブロックを削除。

**H-15. スキーマ管理がアドホックな ALTER TABLE スクリプト頼りでマイグレーション基盤が無い**
- `backend/add_column.py`、`add_check_in_out_dates.py`、`add_folder_table.py`、`migrate_*.py` — Alembic等が無く、`create_all` は既存テーブルにカラムを追加しない。`is_favorite`/`check_in_date`/`folder_id` 等は手動スクリプトを流さないと本番DBに存在せずモデル不一致で実行時エラー。SQLite前提でPostgreSQL本番と乖離。
- **修正**: Alembic を導入し、既存スクリプトを初期マイグレーションに集約。

**H-16. APIキー認証が全キー線形スキャン＋bcrypt照合（性能・DoS）**
- `app/services/api_key_service.py` — リクエスト毎に全アクティブキーを取得し1件ずつ `bcrypt.checkpw`。キー数に比例してbcryptが走り、`/api/v1/ai/generate-plan` が毎回これを通る。`get_api_key_by_hash` は `plain_key=None` を渡すデッドコードで壊れている。
- **修正**: ルックアップ用プレフィックス列（HMAC等の key_id）でインデックス検索してから1件bcrypt照合。

### セキュリティ・インフラ

## 良かった点（土台として機能している箇所）

- SQLは全てSQLAlchemy ORM／パラメータ化 → **SQLインジェクションは検出されず**。
- パスワードは bcrypt ハッシュ。プラン/フォルダ/ユーザーAPIの**取得系は所有者スコープで概ね保護**（他人のプラン取得は404）。管理者チェックは admin API 全エンドポイントで一貫。
- AI生成テキストは React のテキストノードで描画され自動エスケープ（`dangerouslySetInnerHTML` 不使用）。
- 保護エンドポイント（`/api/plans`・`/api/users/me`・`/api/admin/*`）は未認証で正しく **401**。
- バックエンドは実起動可能（`from app.main import app` 成功、`/health`・`/docs` 200、47パス）。
- **外部APIの実キー漏洩は確認されず**（Geminiキーはバックエンド側で正しく秘匿、露出はJWT鍵と管理者PWの2点のみ）。
