# SatoTrip UI リニューアル提案 — 「みちしるべ」

**種別**: デザイン方向性の提案（コードの本適用は含みません）
**成果物**: 本ドキュメント ＋ 単体で開けるモックアップ `docs/ui-redesign/mockup.html`

---

## 0. まず「今やるべきか」

**結論: “デザインを決めて試作する”のは今が適期。ただし全画面の差し替え（本適用）は段階的に。**

理由:

- 機能面はこの数日で大きく安定しました（重大バグ修正・新機能5本・E2E検証・耐障害性強化）。土台が固まった今は、見た目に投資する価値が出てきたタイミングです。
- 一方で、いきなり全画面を差し替えるのは早いです。先に **技術的な前提を1つ**片付けるべきです（下記「前提作業」）。
- したがって推奨は: **①方向性を決める（本提案）→ ②基盤を整える → ③画面単位で段階移行**。本提案は①で、専用ブランチに隔離しているため `main` や進行中の改修に影響しません。

### 前提作業（本適用の前に必要）
- **Tailwind / React の CDN 依存を解消**し、Vite ビルドに一本化する。
  現状 `index.html` は Tailwind と React を外部 CDN から読み込んでいます。E2E 中、隔離環境で CDN がブロックされ「素の見た目」になった根本原因もこれです。デザインシステム（下記トークン）を安定して当てるには、まずビルド時 Tailwind ＋ デザイントークン化が必要です。
- デザイントークン（色・タイプ・角丸・影）を CSS 変数／Tailwind theme に定義し、ライト/ダーク両対応にする。

## 1. コンセプト: 「みちしるべ」

SatoTrip の本当の価値は、**行き先を「歩ける・時間の通ったコース」に変換する**ことです。よくある写真カード中心の旅行 SaaS ではなく、**地図と時刻表でできた“道筋”そのもの**を UI の主役に据えます。

キーワード: **火山と湾の色 / データを語る等幅書体 / 余白の効いたタイムライン**。

鹿児島（桜島・仙巌園・黒豚・白熊・砂むし温泉）という実データの世界観から発想し、量産的な「クリーム地＋セリフ＋テラコッタ」の既視感を意図的に避けています。

## 3. タイポグラフィ

外部フォント CDN に依存せず、**システムフォントスタック**で確実に描画します（CDN ブロック時の崩れを回避）。

- **見出し (Display)**: `Hiragino Sans / Yu Gothic / Noto Sans JP` を **太ウェイト＋詰めトラッキング＋大きめスケール**で。セリフに逃げず、現代的で確信のある和文サンセリフの声。
- **本文 (Body)**: 同上の標準ウェイト、行間 1.7、1行 40〜46字目安。
- **データ (Mono)**: `ui-monospace / SF Mono / Menlo` — **時刻・所要時間・座標・ラベル**を等幅で。これがこの UI の“地図/時刻表”らしさ（差別化点）を担います。例: `09:00 → 11:50 · 20 min · 31.59°N`。

見出しの太い和文サンセリフ × データの等幅、という対比が本デザインの個性です。

## 4. レイアウト原則と主要コンポーネント

- **余白（余白 = 旅情）**: 詰め込まず、1画面1メッセージ。
- **タイムラインが主役**: プラン詳細＝縦タイムライン。時刻（等幅バッジ）→ スポットカード → 移動コネクタ（「↳ 市電 20分」）を破線ルートで繋ぐ。地図の“道筋”のメタファ。
- **カード**: 角丸 14–22px、ハードな枠線ではなく淡い影と細い罫線。
- **カートグラフィック要素**: 破線ルート、ドットのノード、うっすらとした等高線背景、座標/時刻の等幅表記。
- **状態を形で表す**: チップ・バッジ・重要度ストライプ。数値だけに頼らない。
- **モーション**: 控えめ・目的的（ルート線のドロー、カードの浮き）。`prefers-reduced-motion` を尊重。

コンポーネント優先度（作り込む順）:
1. タイムライン（プラン詳細）＝製品の核
2. プラン生成ウィザード（目的地→テーマ→生成の“待ち”体験）
3. スポットカード / お気に入り
4. ナビ・設定・管理画面

## 5. 段階的ロールアウト計画

| フェーズ | 内容 | 影響範囲 |
|---|---|---|
| **P0 基盤** | CDN Tailwind→ビルド化、トークン定義（本提案の色/タイプ）、ダーク対応の土台 | 全体（見た目は現状維持のまま基盤だけ差し替え） |
| **P1 コア画面** | プラン詳細タイムライン＋プラン生成ウィザードを新デザインで | 最も価値が高い2画面 |
| **P2 回遊** | ホーム/スポット一覧/お気に入り/設定 | ユーザー主要導線 |
| **P3 管理** | 管理ダッシュボード等 | 内部向け |

各フェーズを別 PR にし、画面単位で E2E（既に整備した Playwright フロー）で検証しながら進めるのが安全です。

## 6. 非目標・リスク

- 本提案はコードの本適用を含みません（`main` と進行中の改修に非干渉）。
- モックアップはシステムフォント前提の静的 HTML です。実装時は Tailwind theme／CSS 変数へトークンを落とし込みます。
- 絵文字はモックアップのプレースホルダです。実装では登録スポットの実画像／アイコンセットに置き換えます。
